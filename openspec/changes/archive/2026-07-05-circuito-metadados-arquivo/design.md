## Context

`Circuito` (`src/main/java/br/f1mane/entidades/Circuito.java`) é hoje persistido como um único arquivo XML via `java.beans.XMLEncoder`/`XMLDecoder` — `MainPanelEditor.salvarPista()` (linha 2817) escreve, `CarregadorRecursos.carregarCircuito()` (linha 1176) lê. O arquivo mistura metadados leves (`nome`, `ativo`, `noite`, `usaBkg`, `probalidadeChuva`, `velocidadePista`, `ladoBox`, `ladoBoxSaidaBox`, `corFundo`, `corAsfalto`, `corBox1/2`, `corZebra1/2`) com dados pesados (`pista`, `box`, `objetos`, `objetosCenario` e os campos derivados `pistaFull`/`pista1..5Full`/`boxFull`/`box1/2Full`/`pistaKey`/`boxKey`/`escapeMap`/`escapeList`, recalculados por `vetorizarPista()` mas ainda assim persistidos porque `XMLEncoder` serializa qualquer par getter/setter público, ignorando `transient`). Um circuito típico (`catalunya_mro.xml`) tem ~216 mil linhas / ~4,8 MB, dominado por milhares de objetos `No` nos campos derivados. `CarregadorRecursos.circuitoAtivo()` já contorna esse custo hoje com um `BufferedReader` que escaneia as primeiras 200 linhas do XML procurando `property="ativo"` — um workaround frágil (depende da propriedade aparecer perto do início do arquivo) para o mesmo problema que esta mudança resolve de forma estrutural. `properties/circuitos.properties` já existe e já é lido por inteiro por `ControleRecursos.carregarCircuitos()` para montar a listagem de circuitos (`arquivo_mro.xml=NomeExibicao,ciclo`); e `GerenciadorVisual.desenhaMiniCircuito()` (linha 925) já desenha uma miniatura do circuito selecionado usando só `circuito.getPista()` (a lista de nós autorada, não os campos derivados) — hoje obtida chamando `CarregadorRecursos.carregarCircuito()` completo só para isso.

## Goals / Non-Goals

**Goals:**
- Duas gravações por circuito ao salvar: `_mro_meta.xml` (metadados leves + `pista`/`box`) e `_mro.xml` (`objetos`/`objetosCenario`), sem duplicar campos entre os dois.
- Parar de persistir os campos 100% derivados (`pistaFull`, `pista1Full`, `pista2Full`, `pista4Full`, `pista5Full`, `boxFull`, `box1Full`, `box2Full`, `pistaKey`, `boxKey`, `escapeMap`, `escapeList`, `entradaBoxIndex`, `saidaBoxIndex`, `paradaBoxIndex`, `fimParadaBoxIndex`) em qualquer um dos dois arquivos — eles são recalculados por `vetorizarPista()`.
- `_mro_meta.xml` conter `pista`/`box` (a lista de nós autorada pelo usuário, poucas dezenas de itens), suficiente para desenhar uma miniatura do traçado sem ler `_mro.xml`.
- `ativo` sai do XML do circuito e passa a ser o terceiro campo CSV em `circuitos.properties`, único lugar consultado para filtrar circuitos.
- Leitura só de metadados (`circuitoAtivo()`, listagem de circuitos, miniatura) sem tocar no arquivo de objetos nem, no caso de `ativo`, em nenhum XML.
- Carregamento completo (`carregarCircuito()`) continua retornando um único `Circuito` com todos os campos (inclusive `ativo` e os derivados, recalculados/repopulados) populados, sem mudar a assinatura nem o contrato para quem já chama esse método.
- Circuitos existentes (sem `_mro_meta.xml`/formato novo ainda, com `ativo` ainda no XML) continuam carregando corretamente.

**Non-Goals:**
- Não introduzir uma nova classe de domínio (`CircuitoMetadados` ou similar) nem alterar a API pública de `Circuito` — todos os getters/setters existentes (incluindo `isAtivo()`/`setAtivo()`) continuam exatamente iguais, para não propagar mudanças pelos muitos consumidores de `Circuito` no jogo solo, servidor multiplayer e editor.
- Não mudar a lógica de `vetorizarPista()`/`gerarEscapeMap()`/`gerarEscapeList()` em si — só o momento em que rodam (sempre dentro de `carregarCircuito()` agora) e o fato de seu resultado não ser mais gravado em disco.
- Não implementar a miniatura como novo recurso de UI — `desenhaMiniCircuito()` já existe; este change só troca a fonte de dados que ele usa (leitura leve em vez de `carregarCircuito()` completo), sem mudar o desenho em si.
- Não mudar como novos circuitos ganham uma linha em `circuitos.properties` (hoje é um passo manual ao adicionar um circuito novo) — este change só ensina o editor a atualizar o terceiro campo (`ativo`) de uma linha **já existente** ao salvar.

## Decisions

### Split por "zerar e codificar", sem nova classe
Para gerar cada arquivo, criar uma cópia rasa de `circuito` e usar os setters existentes para levar ao valor padrão do bean (`null`/`false`/`0`/lista vazia) os campos que **não** pertencem àquele arquivo, antes de passar para `XMLEncoder`. `XMLEncoder` só emite `<void method="set...">` para uma propriedade quando o valor difere do que uma instância nova (`new Circuito()`) teria — então, ao zerar os campos que não pertencem a cada cópia antes de codificá-la, esses campos simplesmente não aparecem no XML gerado. Resultado: dois arquivos sem sobreposição de conteúdo, usando o mesmo mecanismo de serialização já usado hoje, sem precisar de uma classe nova nem de um `PersistenceDelegate` customizado.
- A cópia usada para `_mro_meta.xml` zera `objetos`/`objetosCenario` (para lista nova vazia, não a mesma referência), `ativo` (volta ao default `false` do bean) e todos os campos derivados (`*Full`/`*Key`/`escapeMap`/`escapeList`/índices de box), mantendo os demais escalares/cores e `pista`/`box`.
- A cópia usada para `_mro.xml` zera os escalares/cores (inclusive `ativo`), `pista`, `box` e todos os campos derivados, mantendo só `objetos`/`objetosCenario`.
- Alternativa descartada — nova classe `CircuitoMetadados` com getters/setters próprios: exigiria que `Circuito` delegasse todos os getters/setters de metadados para um objeto interno (ou duplicasse os campos), tocando a API pública usada em dezenas de arquivos (`ControleRecursos`, `PainelCircuito`, `GerenciadorVisual`, editor, etc.) para nenhum ganho adicional de tamanho de arquivo — a técnica de "zerar e codificar" já produz o mesmo arquivo pequeno sem esse custo.
- Alternativa descartada — manter os metadados também no arquivo de objetos (arquivo de metadados só como adição, sem remover do arquivo grande): rejeitada porque o pedido original é por dois arquivos sem sobreposição ("um arquivo... e *outro* arquivo"), e a técnica de zerar-e-codificar consegue isso sem custo extra de implementação relevante frente a essa alternativa mais simples.

### Campos derivados nunca são persistidos — nem em `_mro_meta.xml`, nem em `_mro.xml`
`pistaFull`, `pista1Full`, `pista2Full`, `pista4Full`, `pista5Full`, `boxFull`, `box1Full`, `box2Full`, `pistaKey`, `boxKey`, `escapeMap`, `escapeList`, `entradaBoxIndex`, `saidaBoxIndex`, `paradaBoxIndex`, `fimParadaBoxIndex` são zerados nas DUAS cópias antes de codificar, então nunca aparecem em nenhum dos dois arquivos. Levantamento nos call sites confirma que isso é seguro: `Circuito.vetorizarPista()` (linha 147) recalcula todos esses campos do zero a partir de `pista`/`box`/`objetos`, e todo consumidor que precisa deles já chama `vetorizarPista()` logo após obter o `Circuito` — `ControleRecursos.carregaRecursos()` (`ControleRecursos.java:140-141`), `MainPanelEditor.carregarCircuitoExistente()` → `vetorizarCircuito()` (`MainPanelEditor.java:1006-1011`), e os três pontos de `LetsRace` (`LetsRace.java:162-164`, `428`, `455`). Ou seja, hoje esses ~4.400 objetos `No` por circuito já são gravados, lidos de volta e imediatamente descartados/recalculados — puro desperdício de I/O e memória de decodificação.
- Alternativa descartada — manter esses campos persistidos no arquivo de objetos (só extrair metadados para o arquivo pequeno): é a leitura mais literal do pedido original, mas ignora que a análise dos call sites mostra que esse dado nunca é reaproveitado do disco; eliminar essa persistência é praticamente sem risco e reduz o arquivo de objetos de ~216 mil linhas para uma fração disso (só `objetos`/`objetosCenario`).

### `carregarCircuito()` chama `vetorizarPista()` antes de retornar
Em vez de depender de cada chamador lembrar de vetorizar a pista antes de usar os campos derivados (padrão espalhado hoje por `ControleRecursos`, `MainPanelEditor` e `LetsRace`), `CarregadorRecursos.carregarCircuito()` passa a chamar `circuito.vetorizarPista()` uma vez, internamente, antes de cachear/retornar. Isso centraliza a garantia (qualquer `Circuito` saído de `carregarCircuito()` já está pronto para uso) e é o que torna seguro parar de persistir os campos derivados — sem essa garantia central, um futuro chamador que esquecesse de vetorizar leria listas vazias.
- As chamadas explícitas existentes a `vetorizarPista()` nos call sites acima passam a ser redundantes (chamar de novo com os mesmos `pista`/`box`/multiplicadores é idempotente) mas continuam inofensivas; removê-las é um cleanup opcional, não necessário para esta mudança.
- Exceção: `MainPanelEditor.vetorizarCircuito()` (chamado pelo editor durante a edição, não só no load) continua existindo e sendo chamado normalmente — ele revetoriza após cada alteração estrutural do usuário (inserir/remover nó), o que é um caso de uso diferente de "carregar do disco".

### Nome do arquivo de metadados: sufixo `_meta` antes de `.xml`, mesmo padrão do jpg
`nmCircuitoXml.replaceFirst("\\.xml$", "_meta.xml")` (ex.: `catalunya_mro.xml` → `catalunya_mro_meta.xml`), no mesmo espírito de `aplicarBackGroundPorConvencao` (que deriva `catalunya_mro.jpg` do nome do XML por convenção, sem gravar o nome do jpg em lugar nenhum). Mantém `_mro.xml` com o nome atual (sem renomear), preservando qualquer referência existente a esse nome (ex.: `circuitos.properties`, `file` escolhido no `JFileChooser` do editor).

### `carregarCircuito()` mescla os dois arquivos num único `Circuito` e popula `ativo` de `circuitos.properties`
Decodifica `_mro_meta.xml` (se existir) num `Circuito` (`circuitoMeta`, com metadados + `pista`/`box`) e `_mro.xml` num outro (`circuitoObjetos`, com `objetos`/`objetosCenario`), depois copia para `circuitoMeta` os setters de `objetos`/`objetosCenario` vindos de `circuitoObjetos` — usando `circuitoMeta` como base final, já que ele é quem tem `pista`/`box`/metadados, que `vetorizarPista()` (chamado em seguida) precisa para recalcular os campos derivados. Em seguida chama `circuitoMeta.setAtivo(CarregadorRecursos.circuitoAtivo(nmCircuito))` para popular `ativo` a partir de `circuitos.properties` (nunca do XML). Se `_mro_meta.xml` não existir (circuito no formato antigo, ainda não migrado), usa só `circuitoObjetos` decodificado de `_mro.xml` (que, por não ter sido migrado, ainda tem `ativo` e tudo mais no arquivo único) e sobrescreve `ativo` do mesmo jeito a partir de `circuitos.properties` — se a migração de `circuitos.properties` já rodou mas a migração de arquivos ainda não (ou vice-versa), `circuitos.properties` continua sendo a fonte de verdade.

### `ativo` sai do XML do circuito e vira o terceiro campo de `circuitos.properties`
Hoje `circuitos.properties` já é uma tabela por circuito (`arquivo_mro.xml=NomeExibicao,ciclo`), já lida por inteiro por `ControleRecursos.carregarCircuitos()` para montar a listagem — adicionar `ativo` como terceiro campo (`arquivo_mro.xml=NomeExibicao,ciclo,ativo`) aproveita uma tabela que já existe e já é o único lugar hoje que precisa saber "quais circuitos existem" antes de decidir carregar algo pesado. Isso também elimina de vez a necessidade de ler qualquer XML (nem o de metadados) só para saber se um circuito está ativo. Levantamento de blast radius: só três usos de `Circuito.isAtivo()`/`setAtivo()` em produção (`MainPanelEditor` — checkbox "Ativo" do editor; `CarregadorRecursos.circuitoAtivo()` — atalho de cache-hit) e um teste; nenhum outro consumidor de `Circuito` lê `ativo` diretamente (todos os seletores de circuito em jogo já passam por `ControleRecursos.carregarCircuitos()`, que já centraliza o filtro).
- `Circuito.isAtivo()`/`setAtivo()` continuam existindo como campo em memória (não removidos da classe), só deixam de ser incluídos em qualquer encode do circuito (zerados nas duas cópias, ver decisão acima) e passam a ser populados a partir de `circuitos.properties` no carregamento.
- Alternativa descartada — remover `isAtivo()`/`setAtivo()` de `Circuito` e mover a responsabilidade para outra classe: o blast radius é pequeno o bastante (editor + um atalho de cache) para não justificar tocar a API pública de `Circuito`; manter os métodos como estão minimiza a mudança no editor (o checkbox "Ativo" continua vinculado a `circuito.isAtivo()`/`setAtivo()` sem alteração de código).

### `circuitoAtivo()` e `ControleRecursos.carregarCircuitos()` leem só `circuitos.properties`
`CarregadorRecursos.circuitoAtivo(String nmCircuito)` substitui o `BufferedReader`/escaneamento de texto por uma leitura de `properties/circuitos.properties`, buscando a entrada `nmCircuito` e lendo o terceiro campo (`Boolean.parseBoolean`); mantém o atalho de cache-hit existente (`bufferCircuitos.get(nmCircuito).isAtivo()`) quando o circuito completo já estiver carregado. `ControleRecursos.carregarCircuitos()` (que já itera o `Properties` inteiro para montar `names[0]`/`names[1]`) passa a ler `names[2]` diretamente no mesmo loop, sem chamar `circuitoAtivo()` por circuito — simplificação natural, já que o arquivo já está todo em memória ali.

### Editor reescreve só o terceiro campo da linha, preservando as demais linhas de `circuitos.properties`
Ao salvar (`salvarPista()`), o editor localiza a linha de `circuitos.properties` cuja chave é o nome do arquivo de circuito sendo salvo, e substitui/acrescenta o terceiro campo CSV pelo valor atual de `ativoCheckBox`, reescrevendo o arquivo linha a linha (não via `Properties.store()`, que reordena e não preserva o formato) — as demais linhas (incluindo `NomeExibicao`/`ciclo` de todos os outros circuitos) são copiadas de volta exatamente como estavam. Se a linha do circuito sendo salvo ainda não existir em `circuitos.properties` (circuito novo, nunca listado), a gravação de `ativo` é pulada (com aviso/log) — criar automaticamente `NomeExibicao`/`ciclo` para um circuito novo não faz parte deste change (ver Non-Goals).
- Alternativa descartada — usar `java.util.Properties.store()` para reescrever o arquivo inteiro: mais simples de implementar, mas reordena as linhas (iteração de `Properties` não preserva ordem de inserção) e escreve um comentário de timestamp no topo, gerando diffs git enormes e ruidosos toda vez que qualquer circuito é salvo.

### `GerenciadorVisual.desenhaMiniCircuito()` passa a usar a leitura leve
`desenhaMiniCircuito()` (linha 925) já usa só `circuito.getPista()` para traçar a miniatura — passa a obter esse `Circuito` parcial (`pista`/`box`/metadados) via a mesma leitura leve de `_mro_meta.xml` usada por `circuitoAtivo()`/listagem, em vez de `CarregadorRecursos.carregarCircuito()` completo, que agora também carregaria `objetos`/`objetosCenario` e vetorizaria a pista à toa só para desenhar uma miniatura de ~40-50 pontos.

### Migração dos circuitos existentes via script único, não via "abrir e salvar" manual no editor
Um pequeno utilitário (executável uma vez, ex. classe `main` temporária ou script) que, para cada `*_mro.xml` em `src/main/resources/circuitos/` no formato antigo: (a) decodifica o circuito, gera as duas cópias zeradas (metadados+pista+box; objetos/objetosCenario) e grava `_mro_meta.xml` e um novo `_mro.xml` (substituindo o arquivo único original); e (b) lê o `ativo` atual do circuito e acrescenta esse valor como terceiro campo CSV na linha correspondente de `circuitos.properties` (mesma técnica de reescrita linha a linha do editor, para não reordenar/perder as demais linhas). Os campos derivados são descartados na migração (não têm equivalente nos novos arquivos, e são recalculados por `vetorizarPista()` no próximo load). Alternativa descartada — pedir para reabrir e salvar cada circuito manualmente no editor: mais trabalhoso e arriscado (o fluxo de salvar já re-vetoriza a pista, podendo introduzir diffs indesejados na geometria existente).

## Risks / Trade-offs

- [Arquivo de metadados ficar desatualizado se alguém editar/gerar o `_mro.xml` fora do editor (ex.: editando o XML manualmente) sem regenerar o `_mro_meta.xml`] → Como o carregamento completo (`carregarCircuito()`) sempre mescla os dois arquivos preferindo os metadados do arquivo dedicado, um `_mro_meta.xml` desatualizado afeta só leituras "rápidas" (`circuitoAtivo()`, listagens) até o próximo `salvarPista()` pelo editor, que sempre regrava os dois juntos e re-sincroniza.
- [Circuito sem `_mro_meta.xml` (não migrado) faz `circuitoAtivo()` cair num caminho mais lento] → Aceitável: cobre só o período de transição até a migração completar; o fallback preserva a corretude (nunca retorna um valor errado), só não ganha a velocidade ainda.
- [Cópia rasa de `Circuito` para zerar campos antes de codificar acidentalmente arrastar referências compartilhadas (ex.: mutar a lista `objetos` da cópia afeta o `circuito` original)] → Ao preparar cada cópia, atribuir listas/mapas **novos** vazios aos campos zerados (não reaproveitar/limpar a lista original), evitando qualquer mutação do `Circuito` em uso pelo editor durante o salvamento.
- [Existir algum consumidor de `Circuito` fora dos call sites já mapeados que leia campos derivados sem antes chamar `vetorizarPista()`] → Mitigado pela decisão de mover a chamada para dentro de `carregarCircuito()`: qualquer `Circuito` que sai desse método (o único ponto de carregamento a partir do disco) já vem vetorizado, então não depende de nenhum chamador individual lembrar disso.
- [Reescrita de `circuitos.properties` pelo editor corromper o arquivo ou perder linhas de outros circuitos, já que agora passa a ser reescrito por código em vez de só à mão] → Reescrita linha a linha preservando texto original (não via `Properties.store()`), testada com um caso que grava e relê o arquivo inteiro conferindo que as demais linhas não mudaram.
- [Migração de `ativo` para `circuitos.properties` divergir do valor migrado nos arquivos `_mro_meta.xml`/`_mro.xml` (ex.: rodar só uma das duas migrações)] → `circuitos.properties` é sempre a fonte de verdade para `ativo` em `carregarCircuito()` (a decisão de merge sempre sobrescreve `ativo` a partir de lá, mesmo se o XML decodificado ainda tiver um valor antigo), então não há divergência possível depois do load.

## Open Questions

- Se cabe também expor um método de listagem em lote (ex.: "metadados de todos os circuitos" para popular um combo sem N leituras individuais) fica a critério da implementação — não é exigido por este change, que cobre só `circuitoAtivo()`/`desenhaMiniCircuito()` como consumidores imediatos da leitura leve.
